# Decisions Reference — Every Design Question, Resolved

**Purpose of this document:** the 7 research files in this folder (`01`–`07`) each answer their own slice of questions in depth. This file pulls every decisive answer into **one flat table** so nothing is left ambiguous, and — critically — **resolves the one real cross-document inconsistency** that surfaced from running 7 independent research agents in parallel. Read this first if you just want "what do we build with," then go to the linked source file for the full reasoning.

## Cross-document inconsistency, resolved: backend language

`03_mobile_tech_stack.md` recommends **Node.js/Express** as the backend (§6, because it shares a language with the eventual verification web dashboard and Firebase Cloud Functions' first-class runtime). `02_cryptography_security_architecture.md` and `05_open_source_connectors.md`, researched independently and in parallel, default their code examples to **JVM libraries** (Bouncy Castle `org.bouncycastle.tsp` for RFC 3161, `java-opentimestamps`, `web3j` for Ethereum/Polygon) because those are the most mature libraries for those specific protocols. Taken together as originally written, this would leave the team with a Node backend that can't directly use the JVM examples. Resolved as follows:

| Function | JVM version (if backend were Kotlin/Java) | **Node.js version (use this — matches the chosen backend)** |
|---|---|---|
| RFC 3161 trusted timestamping | `org.bouncycastle:bcpkix-jdk18on` (`org.bouncycastle.tsp`) | No mature all-in-one npm package exists for RFC 3161 TSP request/response parsing. **Recommendation: skip RFC 3161 for the Node backend** and rely on **OpenTimestamps** (below) as the project's single timestamping mechanism — it's free, has first-class Node support, and Section 4 of `02_cryptography_security_architecture.md` already treats RFC 3161 as optional/parallel to OpenTimestamps, not required. If RFC 3161 is specifically wanted for the "Section 63 hash+algorithm" framing (`06_legal_standards_compliance.md` §1.3), call the FreeTSA endpoint (`http://freetsa.org/tsr`) directly via a raw HTTPS POST of the DER-encoded TSQ — buildable with Node's built-in `crypto`/`https` modules plus a small ASN.1 encoder (`node-forge` can build the request structure), or simplest of all, shell out to the `openssl ts` CLI (already shown working in §4 of `02_cryptography_security_architecture.md`) from a Node child process. Treat as a stretch item either way. |
| OpenTimestamps (Bitcoin anchoring) | `com.eternitywall:java-opentimestamps` | **`opentimestamps`** npm package (github.com/opentimestamps/javascript-opentimestamps; published to npm as `opentimestamps` from v0.4.6 onward — the old `javascript-opentimestamps` package name is deprecated, install the current one: `npm install opentimestamps`). Same stamp/verify/upgrade flow described in `02_cryptography_security_architecture.md` §4, just called from Node instead of the JVM. **This is the corrected, verified package name — use it, not `java-opentimestamps`, since the backend is Node.** |
| Ethereum/Polygon smart-contract calls | `org.web3j:core` | **`ethers.js`** (or `web3.js`) — the standard, first-class Node/JS library for Ethereum-compatible JSON-RPC, deploying/calling Solidity contracts (e.g. against Polygon Amoy testnet), auto-generating typed contract bindings from an ABI. `web3j` was only ever the right pick if the backend were JVM; since `03_mobile_tech_stack.md`'s Node recommendation stands, use `ethers.js` for the `RealityLockAnchor.sol` contract calls described in `02_cryptography_security_architecture.md` §5 — the Solidity contract itself is unchanged, only the calling library differs. |
| SHA-256 hashing (backend-side recompute for verification) | `java.security.MessageDigest` | Node's built-in `crypto.createHash('sha256')` — zero dependency, per `03_mobile_tech_stack.md` §6. |
| ECDSA P-256 signature verification (backend-side) | `java.security.Signature` | Node's built-in `crypto.verify()` / `crypto.createVerify()` with a P-256 public key — zero dependency, per `03_mobile_tech_stack.md` §6. |
| JSON canonicalization (backend-side, to recompute metadata hash) | `io.github.erdtman:java-json-canonicalization` | **`canonicalize`** npm package (implements RFC 8785 JCS for JS) — same RFC 8785 scheme specified in `02_cryptography_security_architecture.md` §1, just the JS implementation instead of the Java one. |
| Play Integrity token decoding | Any JVM Google Cloud client | **`googleapis`** npm package (Google's official Node.js API client), calling `playintegrity.googleapis.com` per the flow in `02_cryptography_security_architecture.md` §3 — Google's Play Integrity REST API is language-agnostic, Node is fully supported. |

**Net effect of this correction:** every crypto/blockchain/timestamping capability described in `02_cryptography_security_architecture.md` and `05_open_source_connectors.md` remains available with the Node.js backend recommended in `03_mobile_tech_stack.md` — nothing needs to be dropped or re-architected, only the specific library name changes from JVM to Node equivalents in the table above. If the team ends up more comfortable in Python/FastAPI (the legitimate alternative noted in `03_mobile_tech_stack.md` §6), the original JVM examples don't apply either — use `rfc3161ng` (RFC 3161), `opentimestamps-client` (Python, per `05_open_source_connectors.md` §4), and `web3.py` (Ethereum) instead.

## Master decision table (all 7 documents, one view)

| # | Question | Decision | Why (one line) | Full detail |
|---|---|---|---|---|
| 1 | Hash algorithm? | **SHA-256** | NIST-approved, zero-dependency, matches what the PPT already committed to; SHA-3/BLAKE3 offer no practical benefit at this scale | `02` §1 |
| 2 | What exactly gets hashed? | **Hash-of-hashes / Merkle root** over media hash + canonicalized metadata hash (2-leaf minimum, 5-leaf designed-for) | Binds media and sensor data together (closes ProofMode's documented gap of signing them separately); enables future selective disclosure | `02` §1, `01` §9 rec. 4 |
| 3 | JSON canonicalization scheme? | **RFC 8785 (JCS)** | Byte-identical hashing of logically-identical metadata; ready-made libraries exist for both Kotlin (`java-json-canonicalization`) and Node (`canonicalize`) | `02` §1 |
| 4 | Signature algorithm? | **ECDSA P-256** (`SHA256withECDSA`) | StrongBox supports P-256 from API 28; does NOT support Ed25519; RSA is ~10x more CPU/battery-expensive | `02` §2 |
| 5 | Where does the private key live? | **Android Keystore**, StrongBox-backed with automatic TEE fallback (`setIsStrongBoxBacked(true)` + catch `StrongBoxUnavailableException`) | Private key never leaves secure hardware; StrongBox is the highest security tier available on commodity Android | `02` §2 |
| 6 | App/device integrity attestation? | **Play Integrity API, Standard request flow** — decode server-side only | SafetyNet is fully shut down (20 May 2025); Play Integrity is the mandatory successor | `02` §3 |
| 7 | Trusted timestamping? | **OpenTimestamps** as primary (free, Bitcoin-anchored, Node-native via `opentimestamps` npm package); RFC 3161/FreeTSA as an optional secondary for Section 63 framing | OpenTimestamps needs no registration/wallet/gas; RFC 3161 gives a named TSA certificate some legal contexts expect | `02` §4, `08` (this file) |
| 8 | Blockchain anchoring approach (the PPT's "Future" item)? | **Polygon Amoy testnet + custom ~20-line `RealityLockAnchor.sol`**, called via **ethers.js** from the Node backend | Fast (~2s blocks) for live demos, free faucet tokens, demonstrates real smart-contract literacy; Hyperledger Fabric is disproportionately heavy for 2 students | `02` §5, `08` (this file) |
| 9 | GPS spoofing detection? | Layered: `isMock()`/`isFromMockProvider()` + `GnssMeasurement` C/N0-AGC analysis + fused-location cross-check + speed/distance plausibility; port `auag0/MockLocationDetector`'s 4-check pattern | No single check is bulletproof (root/Xposed can hide mock flags) — layering + Play Integrity is the realistic mitigation | `02` §6, `05` §5 |
| 10 | Android app architecture? | **MVVM** (ViewModel + Kotlin Flow, not LiveData) | The hard problems live in the data layer (camera/sensor/crypto/sync), not complex UI state — MVI's payoff doesn't match this app | `03` §1 |
| 11 | Camera API? | **CameraX** (`ImageCapture`, `OnImageCapturedCallback`, `CAPTURE_MODE_MINIMIZE_LATENCY`) | <5% perf difference vs. raw Camera2 with far less device-fragmentation work for a 2-person team; `ImageInfo.getTimestamp()` gives a precise, correlatable capture instant | `03` §2 |
| 12 | Location API? | **`FusedLocationProviderClient.getCurrentLocation(PRIORITY_HIGH_ACCURACY)`**, kept "warm" before shutter press | Purpose-built for a single accurate fix; avoids the leak/battery risk of `requestLocationUpdates()` for a one-shot need | `03` §3 |
| 13 | How to correlate sensor/camera timestamps with wall-clock time? | Sensor events and CameraX's `ImageInfo.getTimestamp()` already share the `elapsedRealtimeNanos()` clock base (directly comparable to each other); convert to wall-clock via one `currentTimeMillis() − elapsedRealtimeNanos()/1e6` offset computed at bundle time, store both raw and derived values | Wall clock can jump (NTP/manual changes); the monotonic clock can't, so it's the reliable ordering signal | `03` §4 |
| 14 | Device identity field? | **Locally-generated install UUID** (primary); `ANDROID_ID` optional secondary; **never IMEI** | IMEI is inaccessible to Play Store apps since Android 10; `ANDROID_ID` is a heavier, non-resettable privacy commitment than needed | `03` §5 |
| 15 | Backend framework? | **Node.js + Express** (FastAPI a legitimate alternative if the team is more Python-fluent) | One language across backend + Cloud Functions + any future web dashboard; built-in `crypto` covers everything needed | `03` §6 |
| 16 | Storage/BaaS? | **Firebase** (Firestore + Cloud Storage + Auth + Cloud Functions) | Mobile SDK offline-sync maturity is the deciding factor, reinforcing the Room+WorkManager pattern rather than fighting it; matches what the PPT already committed to | `03` §7 |
| 17 | Offline-first sync pattern? | **Room** local queue (`ProofPackageEntity`, status PENDING/UPLOADING/UPLOADED/FAILED) + **WorkManager** (`NetworkType.CONNECTED`, exponential backoff) | Standard Jetpack-recommended pattern; survives process death/reboot | `03` §8 |
| 18 | Testing strategy? | **JUnit + Robolectric** (shadow classes for `LocationManager`/`SensorManager`) for the crypto/sensor pipeline; **Espresso** only for a handful of UI smoke tests | Puts test effort where bugs are most likely (data/crypto layer) and cheapest to catch, per Google's own testing-pyramid guidance | `03` §9 |
| 19 | Crypto library — Tink vs Bouncy Castle vs raw JCA? | **Google Tink** (`tink-android`) for signing (`PublicKeySign`/`PublicKeyVerify`); raw `MessageDigest` is fine alone for hashing | Tink's high-level primitives make it hard to misuse (no picking a weak algorithm string by accident) — the realistic risk for a student team under deadline pressure | `05` §3 |
| 20 | What AI component to actually build? | **ELA (Error Level Analysis) + EXIF consistency checks** as an "Explainable Authenticity Heuristic," implemented natively in Kotlin — NOT a heavy pretrained deepfake classifier as the primary/only AI work | Threat model (splicing, false re-uploads, edited screenshots) maps to ELA/EXIF far better than to face-swap detectors; fully explainable and defensible in a viva; zero dataset-licensing friction | `04` §6 |

| 23 | Closest open-source prior art to study? | **ProofMode** (`guardianproject/proofmode-android`, GPL-3.0) for the capture→hash→sensor-snapshot→sign pipeline | The single closest existing open-source implementation of this exact idea; well-documented, including its own weaknesses | `01` §3, `05` §1 |
| 24 | Adopt a bespoke proof-package JSON schema, or align with a standard? | Model the "Proof Package" on the **C2PA manifest concept** (assertions + hard-binding hash + signature); use **`contentauth/c2pa-android`** directly if bandwidth allows rather than hand-rolling Keystore code | C2PA is the real industry standard (Adobe/Microsoft/BBC/Truepic); c2pa-android already implements StrongBox→TEE→software fallback signing correctly | `01` §1, §9 rec. 1–2; `05` §2 |

| 32 | Can Reality Lock's output serve as a legal evidence certificate by itself? | **No** — it produces the technical *inputs* (hash value + algorithm + device particulars) a human must still countersign under BSA Section 63's dual-certification requirement (device custodian + independent expert) | Indian law requires natural-person certifiers; software cannot legally self-certify | `06` §1 |

## What's genuinely still open (honest, not padded)

A few items the research explicitly flagged as thin or requiring a judgment call the team — not this research pass — should make:

1. **RFC 3161 on a Node backend has no single clean library** (see the reconciliation table above) — the team should decide whether the "Section 63 hash+algorithm" framing is worth the extra `openssl` shell-out/child-process complexity, or whether OpenTimestamps alone is sufficient for the semester. Recommendation: start with OpenTimestamps only; add RFC 3161 only if time remains.
2. **GPS-spoofing-detection open-source libraries are a thin ecosystem** (`05` §5) — `auag0/MockLocationDetector` is small and low-star; there is no mature, widely-used library at the scale of Tink/ZXing. The team will be doing genuine (small-scale) original engineering here, not just integration.
3. **PRNU on modern phone camera pipelines (HDR/multi-frame noise reduction) is an active, unsolved research problem**, not a solved one (`04` §4) — do not let a professor's question about "why not just use camera fingerprinting" get answered with more confidence than the literature itself has.
4. **No single existing system combines everything Reality Lock proposes** (`01` §8, `07` Synthesized Research Gap) — this is good news for novelty framing, but it also means there is no one reference codebase to copy end-to-end; the team is genuinely synthesizing across ProofMode + c2pa-android + eyeWitness's location-crosscheck idea + OpenTimestamps, not porting one existing app.
5. **Whether to implement the full 5-leaf Merkle tree or the minimal 2-leaf version** (`02` §1) is explicitly left as a time-permitting decision — 2-leaf (`media_hash`, `metadata_hash`) is the required minimum; the 5-leaf version (separate leaves per GPS/timestamp/motion/device-identity) is the "designed-for" architecture to describe in the report even if not fully implemented.
