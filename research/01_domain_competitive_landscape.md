## 1. C2PA and the Content Authenticity Initiative (CAI)

**C2PA** (Coalition for Content Provenance and Authenticity) is a standards body founded in February 2021 by Adobe, Arm, BBC, Intel, Microsoft and Truepic to produce an open, royalty-free technical specification for tracking the provenance (origin and edit history) of digital media — see the [C2PA Specifications site](https://spec.c2pa.org/specifications/specifications/2.4/index.html) and the [C2PA Technical Specification](https://spec.c2pa.org/specifications/specifications/2.4/specs/C2PA_Specification.html). The **Content Authenticity Initiative (CAI)** is the separate, older (2019) cross-industry community that Adobe founded and leads, which builds and maintains the actual open-source implementations of the C2PA spec — see [Content Authenticity Initiative on Wikipedia](https://en.wikipedia.org/wiki/Content_Authenticity_Initiative) and [contentauthenticity.org](https://contentauthenticity.org/). In short: **C2PA writes the standard, CAI writes the code.** The marketing name for the end-user-visible result is **"Content Credentials"** ([Content Credentials — Wikipedia](https://en.wikipedia.org/wiki/Content_Credentials)).

**Technical approach:**
- A **C2PA Manifest** is a digitally-signed, tamper-evident data structure embedded in (or associated with) a media file. It contains **assertions** (claims about origin, actions taken like "cropped"/"edited"/"AI-generated", device info, and optionally the capturing device's certificate) and a **claim signature** over those assertions.
- The manifest container format is **JUMBF** (JPEG Universal Metadata Box Format), assertions are serialized as **CBOR**, and the cryptographic signature uses **COSE** (CBOR Object Signing and Encryption) — the same signing standard used in electronic passports and IoT device attestation.
- **Tamper detection mechanism:** the asset's pixel/byte content is hashed (SHA-256) and that hash is embedded inside the signed manifest as a **"hard binding."** If the file is re-saved or edited, the recomputed hash no longer matches the one in the manifest, and validators flag the credential as invalid or the manifest as detached. This is architecturally the *same core idea* as Reality Lock's "Original Hash ≠ New Hash → Tampering Detected."
- Full detail: [C2PA Implementation Guidance](https://spec.c2pa.org/specifications/specifications/2.4/guidance/Guidance.html), [What is a C2PA Manifest?](https://c2paviewer.com/articles/what-is-c2pa-manifest).

**Open-source SDKs (all real, all live as of 2026):**
- [`contentauth/c2pa-rs`](https://github.com/contentauth/c2pa-rs) — the canonical Rust SDK; implements manifest creation, signing, and validation; dual-licensed MIT/Apache-2.0; exposes a C FFI layer (`c2pa_c_ffi`) that every other-language binding wraps.
- [`contentauth/c2pa-android`](https://github.com/contentauth/c2pa-android) — a **Kotlin wrapper around the Rust C API** for Android, with an example app at [`contentauth/c2pa-android-example`](https://github.com/contentauth/c2pa-android-example) and docs at [opensource.contentauthenticity.org/docs/c2pa-android](https://opensource.contentauthenticity.org/docs/c2pa-android/). Concretely relevant to Reality Lock: it ships `StrongBoxSigner.kt` (hardware-isolated signing keys via Android's **StrongBox Keymaster**, falling back to TEE), `KeyStoreSigner.kt` (software-backed Android Keystore signing), `CertificateManager.kt` (CSR generation) and `WebServiceSigner.kt` (remote/server-side signing). Minimum **Android API 28 (Android 9)**, ES256/P-256 keys only for StrongBox.
- Other official bindings: JavaScript (`c2pa-js`), Node.js, Python (`c2pa-python`), C/C++, and an iOS/Swift library — full roster at [opensource.contentauthenticity.org](https://opensource.contentauthenticity.org/docs/introduction/) and the [contentauth GitHub org](https://github.com/contentauth).
- **Important gap:** the c2pa-android library provides no built-in assertion type for GPS/location or raw sensor data — location/sensor assertions would have to be custom-defined by an implementer (relevant for Reality Lock's GPS+motion-sensor capture, see §9).

**What C2PA does NOT prove** (documented by C2PA's own [Security Considerations](https://spec.c2pa.org/specifications/specifications/1.0/security/_attachments/Security_Considerations.pdf) and independent audits — see §10): it does not fact-check content, does not verify the signer is trustworthy (only that a certificate chain is cryptographically valid), and it explicitly does not prevent a device with a compromised/rooted capture pipeline from signing fabricated content as if it were genuine.

## 2. Truepic — "Controlled Capture"

[Truepic](https://www.truepic.com) is a commercial company (a C2PA co-founder) whose flagship product is **Truepic Lens**, an iOS/Android SDK that **replaces the phone's native camera UI** inside a host app.

**How it works technically:**
1. **Device attestation / pre-authorization** — before any image is captured, Truepic Lens runs OS-level integrity checks (root/jailbreak detection, emulator detection, OS version/patch checks) and only "pre-authorizes" a device to sign media if it passes ([Truepic's New SDK Will Power Trusted Photo Capture](https://www.truepic.com/blog/truepics-new-sdk-will-power-trusted-photo-capture-across-the-internet)).
2. **Capture-time signing** — the SDK captures the frame directly (bypassing the OS camera app and gallery, to prevent a "replay a stored image" attack) and validates the raw pixels, associated metadata, and device identity together, then cryptographically signs the bundle **on-device**, anchored to a certificate issued by Truepic's own Certificate Authority (CA), which is trusted within the C2PA ecosystem ([Truepic Lens Assures a Photo's Authenticity at the Moment of Capture — PetaPixel](https://petapixel.com/2022/05/04/truepic-lens-assures-a-photos-authenticity-at-the-moment-of-capture/), [TIME Best Inventions 2022 writeup](https://www.truepic.com/blog/truepics-authenticating-camera-sdk-recognized-by-times-best-inventions-2022)).
3. **Output format** — the signed result is emitted as a standard **C2PA Content Credential**, so it can be verified by any C2PA-conformant verifier, not just Truepic's own servers ([Truepic Capture product page](https://truepic-website-42ef7248877ffe523886a2e.webflow.io/c2pa/capture)).
4. **Server-side verification** — Truepic also runs a cloud verification service that re-validates the certificate chain, checks it against revocation lists, and can perform deeper forensic checks server-side.

**Open-source status:** **No, Truepic's capture SDK is proprietary and patented** ([Truepic patents on Justia](https://patents.justia.com/assignee/truepic-inc)). Truepic's open contribution is at the *standards* level (co-authoring C2PA) and it *conforms to* the open C2PA manifest format for output, but the actual "Controlled Capture" secure-camera implementation, device-attestation logic, and CA infrastructure are closed and commercial (subscription/enterprise licensing). This is the key structural fact: **Truepic = open output format + closed, patented capture pipeline.**

## 3. ProofMode (Guardian Project / WITNESS / Okthanks) — the closest open-source prior art

[ProofMode](https://guardianproject.info/apps/org.witness.proofmode/) is the single closest piece of open-source prior art to Reality Lock. It has been developed since 2013 by the **Guardian Project**, **WITNESS** (a human-rights video advocacy NGO), and Okthanks ([Combating "Fake News" With a Smartphone "Proof Mode"](https://guardianproject.info/2017/02/24/combating-fake-news-with-a-smartphone-proof-mode/)). Source: [`guardianproject/proofmode-android`](https://github.com/guardianproject/proofmode-android) (GitHub mirror of the canonical GitLab repo), Kotlin/Java, **GPL-3.0 licensed**.

**Exact architecture:**
- **Identity:** on first run the app generates a persistent **OpenPGP keypair** stored locally as the user's "proof identity" (originally just app-sandbox-protected; later releases migrate the key into the **Android Keystore** for hardware backing, per the developers' own response to early criticism — see below).
- **Capture-time behavior:** ProofMode runs as a background service. Whenever a new photo/video appears (originally via filesystem watching, later via Android's native media-store observer API for reliability), it:
  1. Computes a **SHA-256 hash** of the media file.
  2. Takes a simultaneous **snapshot of device sensor/context data** — GPS location, nearby WiFi and mobile network identifiers, altitude, device language, hardware/model info, and more — and writes it to a **CSV file** alongside the media.
  3. **Signs both** the media hash and the sensor-data CSV with the user's **OpenPGP private key**, producing detached `.asc` signature files.
- **Output bundle:** original unmodified media + CSV sensor log + OpenPGP signatures (`.asc`) + a `proof.json` metadata summary — all shareable as a single bundle via Android's share sheet for independent verification by anyone with the public key ([Proofmode + Open Camera — Public Lab](https://publiclab.org/notes/imvec/02-20-2018/proofmode-open-camera)).
- **Verification:** a companion "Proofmode Verifier" checks the OpenPGP signature against the media hash and displays the associated sensor metadata; no proprietary server is required (fully offline-verifiable, which is a real architectural strength over Truepic/Serelay).
- **Timestamping:** roadmap/later releases add **OpenTimestamps** (Bitcoin-blockchain-anchored notarization, via an `android-opentimestamps` module in the repo) so the *existence* of the hash at a point in time can be proven independent of any server or company staying alive.
- **C2PA convergence (as of recent releases):** ProofMode has since integrated the open-source C2PA toolkits — [proofmode.org/c2pa](https://proofmode.org/c2pa) states "ProofMode 3.x is now fully C2PA conformant, supporting the 2.3 specification release," meaning ProofMode now emits standard C2PA manifests in addition to (or instead of) its original OpenPGP/CSV bundle. This is a strong signal that **C2PA has become the de facto interchange format even for grassroots/NGO tooling**, not just Big Tech.

**Documented, publicly-discussed weaknesses (very useful for Reality Lock's own "Limitations" slide):**
- On a **rooted/compromised device**, fake sensor data or fake media can be generated and signed exactly like real data — signing proves "signed by this key," not "this device's sensors were real" ([ProofMode critiques and progress — Guardian Project](https://guardianproject.info/2017/03/30/proofmode-critiques-and-progress/), [Schneier on Security discussion](https://www.schneier.com/blog/archives/2017/03/proof_mode_for_.html)).
- Early versions left the private key/passphrase in process memory without proper scrubbing (a commenter "Thoth" flagged this on Schneier's blog).
- Signing the media hash and the sensor CSV as **separate files** does not stop someone from tampering with *both simultaneously and consistently* before the signing step ever runs (i.e., trust is rooted at "whatever the app read from the OS," not at the camera sensor hardware).
- Embedding GPS + device fingerprint in every photo is itself a **privacy risk** (identifies/deanonymizes at-risk users, e.g. activists) — a tension Reality Lock will also face given its "digital identity protection" use case.
- Guardian Project's own fixes over time: SafetyNet/Play Integrity–style device-attestation checks, Android Keystore migration, and native media-observer APIs — i.e., they converged toward exactly the hardening steps Reality Lock should plan for from day one rather than retrofit.

## 4. Numbers Protocol / Numbers Camera (Capture) / CaptureClub, and Amber Authenticate

**Numbers Protocol** is a Web3/blockchain content-provenance platform. Its consumer capture app is called **Capture** (formerly "Numbers Camera"), with an NFT resale marketplace called **CaptureClub**.
- **Capture mechanism:** at the moment of taking a photo/video, the app collects metadata (timestamp, device info, location), cryptographically signs it with the **user's blockchain wallet key**, and the asset + metadata is registered as a content record identified by a **Numbers ID**, which conforms to **EIP-7053** (an Ethereum content-provenance metadata standard) ([TechCrunch: Numbers Protocol's blockchain camera Capture App](https://techcrunch.com/2021/01/12/numbers-protocols-blockchain-camera-capture-app-safeguards-the-integrity-of-photos/), [All About Capture — Numbers Protocol Medium](https://medium.com/numbers-protocol/all-about-capture-capture-product-suite-capture-cam-8235caf1dd3f)).
- **Storage/anchoring:** content is registered on-chain (Numbers Protocol's own chain plus multi-chain support — Ethereum, Avalanche) and the underlying asset is stored on **IPFS-style decentralized storage**; per current docs the stack combines **"C2PA + ERC-7053"** for cryptographic provenance with **decentralized IPFS storage** ([Capture — Numbers Protocol Documentation](https://docs.numbersprotocol.io/applications/capture/), [captureapp.xyz](https://captureapp.xyz/products/capture-cam/)).
- **Verification:** Numbers Protocol runs an "AI-powered" reverse-image-style engine (nftsearch.site) that checks whether uploaded content matches a registered on-chain original.
- **Open source:** partially — the [`numbersprotocol` GitHub org](https://github.com/orgs/numbersprotocol/repositories) publishes a [TypeScript **Capture SDK**](https://github.com/numbersprotocol/capture-sdk) for developers to integrate registration/minting, but the mobile app itself and the core chain infrastructure are Numbers Protocol's commercial product, not something a student team can simply fork wholesale.
- Note the **NFT/monetization framing** is a real product-market difference from Reality Lock: Numbers Protocol is built around *making content into a tradeable, ownable asset*, whereas Reality Lock is built around *proving an event happened* — the crypto-asset layer is mostly irrelevant baggage for Reality Lock's use cases (legal evidence, insurance, industrial safety).

**Amber Authenticate** (by Amber Video) targets a different niche: continuous video streams (e.g. police body cameras), not single-photo capture.
- It runs **in the background during recording** and, at fixed intervals, computes cryptographic hashes ("fingerprints") of chunks of the video stream and writes those hashes to a **public blockchain via smart contracts**, so that even after the footage is later cut/edited/recombined, each surviving segment's chain-of-custody can be traced back to its original hash-anchored fragment ([Amber Video: How blockchains can be used in authenticating video](https://medium.com/amber-video/how-blockchains-can-be-used-in-authenticating-video-and-countering-deepfakes-25d596ad7a5), [CPI OpenFox: How Blockchain Secures Chain of Custody](https://www.openfox.com/news/how-blockchain-secures-chain-of-custody-in-an-era-of-ai-deepfakes/)). Amber's companion product **Amber Detect** uses AI/signal-processing to flag maliciously altered audio/video after the fact — i.e. Amber bundles both a *provenance* tool and a *forensic detection* tool, which is a useful two-pronged model to note (though not one Reality Lock needs to copy at semester scope).
- Not open source; enterprise/government sales model (law enforcement, government agencies).

## 5. Serelay — "digital fingerprint at time of capture"

[Serelay](https://www.trublo.eu/2021/02/24/serelay-start-taking-verifiable-trustable-photos-with-your-mobile-phone-today/) was a UK (Cambridge-area) startup whose secure camera app computed a **cryptographic "digital fingerprint"** — reportedly **300–500 data points** derived mathematically from the image (not a simple whole-file hash) — at the exact instant of capture ([311 Institute: DeepFake busting app spots even a single pixel out of place](https://www.311institute.com/deepfake-busting-app-spots-even-a-single-pixel-out-of-place/)).
- **Mechanism:** rather than hashing the whole file (which breaks on any re-compression/re-encoding), Serelay's approach computed ~100 mathematical values summarizing regions/features of the image, allowing it to detect and **localize** roughly *which part* of an image had been edited later, even after the image had been through compression pipelines like social-media re-uploads — a materially different and more forensically useful approach than a plain SHA-256 hash-mismatch, which only tells you "something changed," not "where."
- Distributed as free Android/iOS apps; verification (server-side, comparing the stored fingerprint to a freshly-recomputed one from a submitted image) was Serelay's core commercial service, sold to media/publishing and insurance industries ([Businesswire: Serelay Content Authenticity Verification Now Compatible with Adobe Apps](https://www.businesswire.com/news/home/20210318005018/en/Serelay-Content-Authenticity-Verification-Now-Compatible-with-Adobe-Apps)).
- **Not open source** — proprietary algorithm and closed server-side verification.
- **Critical, decision-relevant fact: Serelay Limited was formally dissolved on 2 March 2025** ([UK Companies House record](https://find-and-update.company-information.service.gov.uk/company/10672861); founder's directorship ended August 2023 per [Tracxn profile](https://tracxn.com/d/companies/serelay/__dBj6r4k3dm3QkWrW3aJdqjPqAXvKN3xzY5HkzZuaxIc)). This is real, current evidence that even a well-funded, patent-holding, decade-old startup in exactly this space **did not survive commercially** — a cautionary data point for the "Key Recommendations" section below regarding scope realism.

## 6. Starling Lab (Stanford/USC) and eyeWitness to Atrocities (IBA)

**Starling Lab for Data Integrity** is a joint Stanford/USC academic research lab (cryptography + distributed systems + journalism/human-rights experts) building "authenticity-by-design" infrastructure ([starlinglab.org](https://www.starlinglab.org/), [USC/Stanford launch coverage — VentureBeat](https://venturebeat.com/consumer/usc-and-stanford-launch-starling-lab-to-protect-human-rights-with-decentralization)).
- Their methodology is the **"Starling Framework"**: a three-stage pipeline of **secure capture → decentralized/redundant storage → independent verification**, applied end-to-end across an asset's lifecycle ([The Starling Lab Framework — FFDW](https://ffdweb.org/blog/the-starling-lab-framework)).
- **Real deployments:** worked with **Reuters** and Canon on a pilot ("Photographic Archive of Trust") that securely captured, stored (across multiple redundant systems including Adobe's CAI, **Filecoin**, and **Hedera Hashgraph**), and verified newswire photographs across all 78 days of the 2020–21 US presidential transition ([Reuters press release](https://www.thomsonreuters.com/en/press-releases/2023/august/reuters-new-proof-of-concept-employs-authentication-system-to-securely-capture-store-and-verify-photographs)). Starling Lab has also documented Ukraine war-crimes evidence and Hong Kong protest documentation with similar methodology.
- **Open-source tooling:** published under the [`starlinglab` GitHub org](https://github.com/starlinglab), notably [`authenticated-attributes`](https://github.com/starlinglab/authenticated-attributes) (a key-value store, built on Hyperbee, for storing authenticated metadata attributes tied to assets, with an HTTP API) and [`integrity-backend`](https://github.com/starlinglab/integrity-backend) (a configurable backend server for registering and processing authenticated assets in their "Integrity" framework). These are real, inspectable, MIT/Apache-style-licensed reference implementations of a capture→store→verify pipeline — arguably the best *architectural* reference for Reality Lock's own backend design, even though they are research-grade rather than production-polished.
- **Multi-anchor redundancy is their signature idea:** rather than trusting one blockchain or one cloud provider, Starling Lab deliberately anchors hashes/proofs to *multiple, independent* systems (Filecoin **and** Hedera **and** conventional cloud) so that no single vendor's failure or shutdown (cf. Serelay above) invalidates the proof.

**eyeWitness to Atrocities** is a mobile app created in 2015 by the **International Bar Association (IBA)** (now IBA + LexisNexis) specifically to gather footage of international crimes that is admissible in court ([IBA: ten years of eyeWitness](https://www.ibanet.org/IBA-founded-eyeWitness-to-Atrocities-marks-ten-years-of-capturing-evidence-of-human-rights-crimes)).
- **Metadata captured at time of recording:** GPS latitude/longitude *with accuracy figure*, nearby WiFi network identifiers, nearby cell-tower identifiers, GPS-derived date/time, **and** independent device date/time — i.e. **three independent sources used to cross-check location and time** (GPS, WiFi, cell tower), specifically so a single spoofed GPS signal cannot silently fabricate a location ([eyeWitness: Using metadata](https://www.eyewitness.global/Using-metadata)).
- **Hash & chain of custody:** a hash value ("digital fingerprint") is generated automatically the instant the footage is captured; the app **refuses to accept media imported from outside the app itself** (blocking a "sign an already-tampered file" attack vector that plagues ProofMode-style designs); footage cannot be exported/shared locally — it must be uploaded, encrypted, to eyeWitness's secure server first, after which it is locked in segregated, PIN-protected storage separate from the phone's normal photo gallery, and every subsequent access is logged, preserving legal chain-of-custody ([eyeWitness FAQs](https://www.eyewitness.global/FAQS)).
- As of 2025 the app has recorded over **53,000 items** of documented evidence, some of which have already been used in real war-crimes court cases in Ukraine ([dev.ua coverage](https://dev.ua/en/news/53000-tsyfrovykh-dokaziv-deiaki-z-iakykh-vzhe-v-sudakh-yak-zastosunok-eyewitness-to-atrocities-fiksuie-voienni-zlochyny-rf-i-zakhyshchaie-tykh-khto-nym-korystuietsia-1739962240), [ijnet.org story](https://ijnet.org/en/story/how-eyewitness-atrocities-empowers-journalists-record-evidence-international-crimes)).
- Not open source; run as a nonprofit legal-tech service, not a published SDK.

## 7. InVID/WeVerify — a fundamentally different category

[InVID-WeVerify](https://weverify.eu/verification-plugin/) is a free browser extension (originally an EU Horizon 2020 research project, "InVID," later extended as "WeVerify" and now "VeraAI") used by journalists/fact-checkers/OSINT investigators. It is **not** a capture-time proof system at all — it is a **post-hoc forensic investigation toolkit** for media that is *already circulating online* and whose origin is unknown ([Fake news debunker extension listing](https://chromewebstore.google.com/detail/fake-news-debunker-invid/mhccpoafgdgbhnjfhkcmgknndkeenfhe)).

Its feature set: reverse-image search across Google/Bing/Yandex/Baidu/TinEye, video keyframe extraction (so keyframes can themselves be reverse-searched), EXIF metadata reading, in-browser **Error Level Analysis (ELA)** and copy-move forgery detection, OCR, and deepfake/voice-clone screening heuristics ([InVID-WeVerify tool card](https://denis989.github.io/ipmr-toolkit/tool-cards/invid-weverify/)). Critically, its own documentation is explicit that it "does not detect fake news automatically" and gives "no simple true/false verdict" — it's a workflow aid for human investigators, with ~57k weekly active users.

**Why this matters for Reality Lock's literature survey:** InVID/WeVerify represents the *opposite end of the pipeline* from C2PA/Truepic/ProofMode/Reality Lock. Those systems try to **prevent** the ambiguity by locking in cryptographic proof **at the moment of capture**, when the ground truth is knowable. InVID/WeVerify instead tries to **detect after the fact**, working backward from an already-published, provenance-less file using forensic/statistical inference — which is inherently probabilistic and beatable by good-enough fakes. This is the single cleanest way to explain, on a slide, why "capture-time" systems like Reality Lock are a structurally different (and arguably stronger) approach than "AI deepfake detection" — which is exactly the fourth bullet already listed on the Literature Survey slide (Slide 7) as an existing but limited technology.

## 8. Android-specific open-source prior art (direct GitHub search results)

Beyond ProofMode, several Android-specific projects sit very close to Reality Lock's exact concept:

| Project | What it is | License / status |
|---|---|---|
| [**CameraV / InformaCam**](https://github.com/guardianproject/CameraV) (Guardian Project) | The direct predecessor to ProofMode. Captures photo/video plus a rich sensor snapshot (compass, ambient light, temperature, GPS, and more), signs/encrypts using **OpenPGP + IOCipher + Tor**, and lets the user "**notarize**" media by broadcasting the unique fingerprint ID + sensor metadata via email/SMS/social media as an independent, out-of-band time-anchor. Documented in full at the [InformaCam Guide](http://guardianproject.github.io/informacam-guide/en/InformacamGuide.html). | Open source, freely licensed (archived/superseded by ProofMode, but still browsable and a legitimate architecture reference) |
| [**ProofShot**](https://play.google.com/store/apps/details?id=com.proofshot.app) | Android app that records GPS **before, during, and after** the shot (to prove *continuity of presence*, not just a single instant — directly useful precedent for Reality Lock's "location plausibility" verification goal), embeds SHA-256-hashed signature data with device info, and lets users selectively grant GPS/camera/mic permissions with full local delete. | Closed-source consumer app, but a clean, current (still-live) product-design precedent |
| [**Trustamp**](https://trustampcamera.com/) | A live 2025/2026-era Android camera app aimed at gig-economy proof-of-delivery/inspection use cases. Issues a **server-witnessed "witness code" *before* the shutter fires** (so the capture window can't be backdated even if the phone's clock is manipulated), locks GPS at country/city/street granularity, and optionally seals the image with a real **C2PA Content Credential** on-device; publishes a public verification link (`verify.trustampcamera.com`) that survives even after social platforms strip EXIF/metadata on re-upload. | Not open source; closest *live commercial analog* to Reality Lock's own use-case framing (delivery/insurance/field inspection) |
| [**VeraSnap**](https://dev.to/veritaschain/verasnap-building-a-cryptographic-evidence-capture-app-for-android-with-kotlin-camerax-and-3p2f) (VeritasChain) | A described (in-progress) Android reference implementation of a **"Content Provenance Protocol" (CPP)**, built on **CameraX + Kotlin + Jetpack Compose + Hilt + Room**. Extremely close technically to what Reality Lock is proposing: SHA-256 content hash, **ES256/ECDSA-P256** signature, hash-chaining between successive events, mandatory **RFC 3161 Time-Stamp-Authority countersigning of the hash only** (never the photo itself) for independent offline verification years later, **Android StrongBox/TEE-backed key storage** via a `KeystoreManager`, optional **BIOMETRIC_STRONG** gating of captures, and a "screen re-photograph" detector using depth-sensor variance/flatness heuristics (to catch someone photographing a screen instead of reality). | The **CPP protocol spec** is published open source at [`github.com/veritaschain/cpp-spec`](https://github.com/veritaschain/cpp-spec); the VeraSnap **app itself** is proprietary in development, no public app repo found |
| [**OTScam**](https://github.com/lvaccaro/OTScam) | Minimal open-source Android app: timestamps a photo using **OpenTimestamps** (Bitcoin-anchored) directly from the camera in the background. | Open source (small, unmaintained-looking, but a clean minimal reference for OpenTimestamps-on-Android integration) |
| [`nesl/AndroidCameraNTP`](https://github.com/nesl/AndroidCameraNTP) | Academic (UCLA NESL) prototype: basic Android camera app that also exports **NTP-synchronized timestamp data** with video, addressing the "device clock can't be trusted" problem via network time sync rather than blockchain/TSA. | Open source academic prototype |
| [`contentauth/c2pa-android`](https://github.com/contentauth/c2pa-android) + [`c2pa-android-example`](https://github.com/contentauth/c2pa-android-example) | Covered in §1 — the official, production-grade, actively maintained C2PA Android SDK with StrongBox signing example app. | Open source (Apache-2.0/MIT family) |

**Direct GitHub topic/keyword searches** for `"tamper evident camera android"`, `"trusted timestamp camera app"`, and `"proof of capture android"` surface no other maintained, well-known project beyond the ones above (results otherwise return unrelated packet-capture tools, generic app-tamper-*detection* libraries like [`mukeshsolanki/Android-Tamper-Detector`](https://github.com/mukeshsolanki/Android-Tamper-Detector) that detect if *the app's own APK* has been modified rather than whether *captured media* is authentic, and old, abandoned student projects). **Conclusion: there is no single dominant open-source "reference app" for this exact idea on Android** — the space is real but fragmented across NGO tooling (ProofMode/CameraV), a big-tech standard (C2PA/c2pa-android), and several small/commercial/in-progress apps (ProofShot, Trustamp, VeraSnap). This is good news for the project's novelty framing and bad news in the sense that there is no single "gold standard" codebase to simply study end-to-end — the team will need to synthesize ideas from several of these.

## 10. Academic/industry critiques of blockchain-based "immutable evidence" claims

This is the exact gap the Literature Survey slide (Slide 7: *"Blockchain Evidence Storage → Provides data immutability but does not verify real-world event authenticity"*) names but does not elaborate. The research literature is unambiguous and consistent:

1. **The "Garbage In, Garbage Out" (GIGO) problem.** Peer-reviewed research on IoT+blockchain supply chains states plainly that blockchain does not resolve conventional data-quality issues — "data on a blockchain may simply be immutable garbage" because "blockchain technology does not offer any mechanism to verify the correctness of the data submitted by a user" ([Garbage in garbage out: The precarious link between IoT and blockchain in food supply chains — ScienceDirect](https://www.sciencedirect.com/science/article/abs/pii/S2452414X21000595)). The same paper argues GIGO problems "require solutions outside of the blockchain" — specifically robust methods at the **physical-to-digital interface** (i.e., trusted sensing/attestation), which is exactly the layer Reality Lock's "Security Engine" and sensor-capture stage need to be honest about.
2. **Blockchain secures data *after* it's written, not *at the point of origin*.** A broader survey on blockchain in supply chains confirms: "IoT devices may be faulty or inappropriately deployed; they may be tampered with; communications may be intercepted and data veracity compromised. Blockchain mechanisms do not guarantee the trustworthiness of data at the source" ([Blockchain in Supply Chain: Opportunities and Design Considerations — arXiv](https://arxiv.org/pdf/2108.12032)).
3. **Blockchain notarization proves "existed and unaltered since T," not "true."** Research on blockchain-based social-media notarization states this explicitly: nodes in a blockchain network "do not have the ability to judge the authenticity of input data" — a notarized hash proves a file existed in that exact byte-form at a timestamp, and has not been altered *since*, but says nothing about whether the photographed event actually happened as depicted. This is the precise logical distinction Reality Lock's Verification Module needs to be careful never to overstate: "Hash Match" + "Signature Verify" + "Time Validate" + "Location Verify" together prove **chain-of-custody integrity from the moment of capture**, not **that the depicted event is real** (an AI-generated image, signed and hashed the instant it's generated, passes every one of those checks perfectly).
4. **The sensor-trust / "root of trust" problem is the crux, and it's an active, unsolved research area.** Independent security research on C2PA specifically — [Verifying Provenance of Digital Media: Why the C2PA Specifications Fall Short (arXiv, 2026)](https://arxiv.org/html/2604.24890v1) and the companion [IACR ePrint security analysis](https://eprint.iacr.org/2026/804) — documents concrete, real-world failures: conforming C2PA validators are **not required** to check certificate revocation (a Nikon Z6 III camera certificate revoked in November 2025 was still being accepted as valid by one major validator six months later); nothing in the signed manifest cryptographically binds the *timestamp* itself, so a manifest's claimed capture time can be silently altered without any validator flagging it; different "conformant" validators disagree on the same file (one flags AI-generated, another doesn't); and on at least one real device (Pixel 10 Pro) GPS coordinates were found to sit in a metadata region **excluded** from the manifest's hard-binding hash, meaning an attacker can rewrite the claimed capture location without invalidating the signature at all. A separate technical/privacy review from the [World Privacy Forum](https://worldprivacyforum.org/posts/privacy-identity-and-trust-in-c2pa/) independently confirms: "C2PA doesn't fact-check content or verify creator trustworthiness; it merely validates cryptographic signatures," and documents that researchers have already produced forged content with technically-valid C2PA manifests using standard tools "in minutes."
5. **GPS/sensor spoofing is a solved *attack*, not a solved *defense*.** Academic work on camera-sensor fingerprinting for geo-position verification exists precisely because GPS metadata alone is trivially spoofable ([Geographical position spoofing detection based on camera sensor fingerprint — ScienceDirect](https://www.sciencedirect.com/science/article/abs/pii/S104732032100211X)), and conversely, camera-sensor-pattern-noise fingerprints (long considered a robust forensic anchor) have themselves been shown to be spoofable via GAN-injected synthetic camera traces ([SpoC: Spoofing Camera Fingerprints — arXiv](https://arxiv.org/pdf/1911.12069)). **There is no known sensor signal on a commodity Android phone that is unconditionally un-spoofable**; every layer (GPS, camera PRNU fingerprint, device clock, accelerometer) can be attacked given a sufficiently motivated, sufficiently resourced adversary (typically: a rooted device or a hardware GPS spoofer). This is why hardware-backed key attestation (TEE/StrongBox — see §1, §8) matters more than any one sensor reading: it raises the *cost* of forgery from "software-only" to "requires breaking real secure hardware," which is the most realistic bar a semester project can defensibly claim.

**Bottom line for Reality Lock's own Literature Survey slide:** the correct, precise claim is not "blockchain/hash+signature = proof the event is real." The correct, precise, and defensible claim is: **"a hash + digital signature + hardware-backed key + independent timestamp anchor proves that this exact bundle of media+metadata existed, unaltered, at time T, on a specific attested device"** — which shifts the remaining trust question from "was the file edited after capture?" (which cryptography answers well) to "was the *device's sensor pipeline* itself honest at capture time?" (which cryptography alone cannot answer, and which is the actual open research problem across every system surveyed above, not a solved one).

## Key Recommendations for Reality Lock

1. **Adopt a C2PA-style manifest as your proof-package schema, not a bespoke JSON format.** Don't invent a new container structure from scratch — model the "Proof Package" (Slide 9) directly on the C2PA manifest concept (assertions + hard-binding hash + COSE-style signature), but add first-class assertion types for GPS, accelerometer/gyroscope, and environmental sensors, since C2PA itself has no standard slot for these. This gives the project a credible "we align with an actual industry standard" claim while still leaving genuine room for original contribution (the sensor-fusion assertions).

2. **Use `contentauth/c2pa-android` directly rather than hand-rolling Keystore/signing code.** It already implements exactly the StrongBox → TEE → software fallback chain Reality Lock's "Security Engine" needs, with a working CSR/certificate enrollment flow. Re-implementing this from scratch in a two-student semester timeline is wasted effort when a maintained, permissively-licensed (MIT/Apache) library does it correctly already. Budget the saved time toward the sensor-fusion and verification-module work instead, which is where the actual novelty lives.

3. **Copy ProofMode's "sign everything, ship the bundle, verify fully offline" philosophy over Truepic/Serelay/eyeWitness's "call our server" model.** Serelay's dissolution (§5) and eyeWitness/Truepic's dependency on a single company's backend are the clearest cautionary tales in this entire survey: a proof system whose verification *requires* a specific company's server to still exist in ten years is a bad design for "legal evidence" and "insurance claims" use cases that may need to be checked years later. Reality Lock should make offline verification (signature + hash-recompute, no network call required) the primary path, with cloud storage as a *convenience/backup*, not a *dependency*.

4. **Do close ProofMode's own documented gap: cryptographically bind sensor data to media, don't just co-sign two separate files.** Concretely: compute the media hash, then compute a combined hash over `(media_hash || canonicalized_sensor_json)`, and sign *that* combined hash. This one change is cheap to implement and directly answers a real, publicly-documented criticism ("I can sign whatever sensor snapshot I want with whatever picture I want" — Schneier's commenters, §3) of the nearest open-source prior art.

5. **Cross-check location with more than raw GPS, following eyeWitness's model.** At minimum, capture GPS **and** note nearby WiFi SSIDs/cell-tower IDs (both readable via standard Android APIs without special permissions beyond location) so the Verification Module's "Location Verify" step has more than one independently-spoofable signal to check for consistency. This is a small implementation cost with an outsized credibility payoff, and it's exactly what a real legal-evidence-grade system (eyeWitness) does.

6. **Use a free, public timestamp anchor (OpenTimestamps or an RFC 3161 TSA) for the hash only — do not build custom blockchain infrastructure.** Both ProofMode (via its `android-opentimestamps` module) and the VeraSnap/OTScam projects converge on the same lightweight pattern: send only the hash (never the media) to a free Bitcoin-anchored timestamping service or a public RFC 3161 Time-Stamp Authority. This delivers the "Blockchain Integration" line already on Slide 8's roadmap as a **realistic, no-cost, no-custom-infrastructure "Future" milestone**, rather than standing up a private chain or smart contracts, which is disproportionate to a semester timeline and would not actually improve the proof's strength over a free public timestamp anchor.

7. **Explicitly reject the NFT/monetization layer that Numbers Protocol/CaptureClub bundle in.** It is scope-inflating, legally/ethically irrelevant to Reality Lock's stated use cases (legal evidence, insurance, emergency response, journalism, industrial safety, digital identity), and would dilute the project's credibility as a serious authenticity tool rather than a crypto-collectibles app. Cite Numbers Protocol only for its ERC-7053 metadata-standard idea, not its product model.

8. **On the "Verification Module," state the trust boundary honestly rather than overclaiming "proof the event is real."** Per §10, phrase the Authenticity Result as certifying **chain-of-custody integrity from the point of capture** (unaltered since time T, on an attested device, at a GPS-plausible location), not as an unconditional guarantee that "this really happened." This precise framing is itself a differentiator: it demonstrates the team understands the sensor-trust/oracle problem that every commercial competitor surveyed here (Truepic, Serelay, Numbers Protocol, Amber) glosses over in their marketing but that the academic literature (§10) is now actively calling out as a real, unresolved weakness even in C2PA. A Zeroth Review that names this limitation explicitly, and explains the hardware-attestation mitigation (StrongBox/TEE), will read as materially more rigorous than the source material's current one-line "Research Gap" bullet.

9. **For prototyping priority order given a two-person, one-semester team:** (1) CameraX-based custom capture screen that blocks gallery import, mirroring eyeWitness/Truepic's "in-app capture only" rule; (2) SHA-256 hash + combined-sensor-hash binding, mirroring recommendation #4; (3) `c2pa-android` StrongBox/Keystore signing integration; (4) Firebase/cloud storage of the bundle; (5) offline verifier module (recompute hash, check signature, check GPS/WiFi/cell plausibility); (6) OpenTimestamps hash anchoring as a stretch/"Future" goal. This ordering front-loads the pieces with the most existing, reusable open-source code (CameraX, c2pa-android) and defers the piece with no existing reusable Android library (multi-signal location plausibility) to where it can get the most design attention.
