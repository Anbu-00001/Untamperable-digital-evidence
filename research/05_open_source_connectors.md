| **Numbers Protocol — Capture Cam** | [github.com/numbersprotocol/capture-cam](https://github.com/numbersprotocol/capture-cam) | GPL-3.0 | "A photo-sharing app with only verifiable photos and videos" — Ionic/Capacitor + TypeScript hybrid app (26★, actively released, v0.103.2 as of Sep 2025) from Numbers Protocol, which anchors media provenance to their own "Nyoiwa"/Numbers blockchain layer and integrates C2PA. | A working, still-maintained example of a **cross-platform (not native-Android)** capture app that does capture → hash → provenance-record → blockchain-anchor, i.e. closest to Reality Lock's end-to-end vision including the blockchain leg. Worth a screenshot/UX reference for the verification-badge UI and as evidence the market niche is real and active. |

**Note on VeraSnap**: search turned up "VeraSnap" (VeritasChain), a commercial Android/iOS app doing SHA-256 + digital signatures + RFC3161 timestamping + depth-sensor liveness checks — conceptually very close to Reality Lock. The **app itself is closed-source** (App Store / Play Store product), but the underlying protocol it implements, the **Content Provenance Protocol (CPP)**, is published as an open IETF Internet-Draft and spec repo at [github.com/veritaschain/cpp-spec](https://github.com/veritaschain/cpp-spec). Useful as a citable open **specification** for the literature survey, not as reusable code.

## 3. Cryptographic libraries for Android/Kotlin

| Name | GitHub/URL | License | What it does | How Reality Lock would use it |
|---|---|---|---|---|
| **Google Tink** (`tink-android`) | [github.com/tink-crypto/tink-java](https://github.com/tink-crypto/tink-java) (org: [github.com/tink-crypto](https://github.com/tink-crypto); Maven artifact `com.google.crypto.tink:tink-android`) | Apache-2.0 | Google's crypto library (294★ on the Java repo) designed specifically so misuse is hard: high-level "primitives" (`Aead`, `PublicKeySign`/`PublicKeyVerify`, `HybridEncrypt`) instead of raw algorithm parameters, safe defaults baked in, keys managed via `Keyset` objects. | Use `PublicKeySign`/`PublicKeyVerify` (ECDSA) to sign the SHA-256 hash of each proof package, and `Aead` if any local data needs at-rest encryption. Two students with a one-semester timeline are far less likely to shoot themselves in the foot (e.g. reusing an IV, picking ECB mode) than with raw JCA. |
| **Bouncy Castle** (`org.bouncycastle`) | [github.com/bcgit/bc-java](https://github.com/bcgit/bc-java) | MIT-style (Bouncy Castle License) | The most complete JCA/JCE crypto provider for the JVM (2.7k★) — full algorithm suite (RSA, ECDSA, EdDSA, X.509 cert handling, OpenPGP, TLS). Historically needed on Android via the "Spongy Castle" repackaging ([github.com/rtyley/spongycastle](https://github.com/rtyley/spongycastle)) to avoid clashing with Android's bundled, older BC provider, though modern Android/AndroidX projects can often use `bcprov-jdk18on` directly. | Only reach for this if Tink's primitive set is missing something specific (e.g. a particular PGP-compatible signature format to interop with ProofMode's own `.asc` signatures, or a specific curve). It is powerful but low-level — every algorithm/mode/padding choice is manual, which is exactly the kind of decision a 2-student team is most likely to get subtly wrong. |
| **Raw JCA** (`java.security`, `javax.crypto`) | N/A — built into the JDK/Android SDK, no external repo | N/A (platform API) | The standard Java Cryptography Architecture: `MessageDigest.getInstance("SHA-256")`, `Signature.getInstance("SHA256withECDSA")`, `KeyPairGenerator`, and (on Android) the hardware-backed `AndroidKeyStore` provider. | Suitable for the SHA-256 hashing step alone (`MessageDigest` is simple and hard to misuse). For signing, JCA's flexibility is also its risk — it's easy to pick a legacy/weak algorithm string and not notice. Best used *underneath* Tink or in combination with `AndroidKeyStore` for hardware-backed key storage, not as the sole crypto layer. |

**Comparison & recommendation**: Raw JCA is the leanest (no new dependency) but offers no guardrails — a wrong string literal (`"SHA1withRSA"` vs `"SHA256withECDSA"`) silently compiles and silently weakens security, which is a realistic failure mode for a student project under deadline pressure. Bouncy Castle is the most powerful and the most "citable" in an academic report, but its API surface requires understanding padding/mode/curve choices explicitly — appropriate for a crypto-focused elective, overkill and riskier for an app-dev project where crypto is one subsystem among many (capture, GPS, cloud, UI). **Tink is the better fit**: Google built it precisely to remove the footguns JCA and Bouncy Castle both expose, its `PublicKeySign`/`PublicKeyVerify` API maps 1:1 onto "sign this hash, verify this signature," and it still uses `AndroidKeyStore` under the hood for hardware-backed keys when configured that way.

| **MockLocationDetector** (auag0) | [github.com/auag0/MockLocationDetector](https://github.com/auag0/MockLocationDetector) | MIT | Small Android library/app (48★) implementing four independent mock-location checks: (1) `AppOpsManager` scan for any installed app holding `ACCESS_MOCK_LOCATION`/mock-location app-op, (2) reading the deprecated `Settings.Secure` `mock_location` flag for legacy-OS coverage, (3) inspecting the `Location` object's own mock flags (the modern `isFromMockProvider()` equivalent, cross-checked), and (4) basic reflection/hook-tampering detection on the `Location` class fields. | Directly portable reference for a `LocationIntegrityChecker` class: layering all four checks (rather than relying on `isFromMockProvider()` alone) meaningfully raises the bar against simple GPS-spoofing apps, which is exactly the gap the assignment brief calls out. |

| *(Cross-platform reference only, not directly reusable)* **fake_gps_detector** | [github.com/andiksetyawan/fake_gps_detector](https://github.com/andiksetyawan/fake_gps_detector) | Check repo (typically MIT for this class of plugin) | A **Flutter** plugin doing emulator + mock-location detection. | Not usable as-is in a native Kotlin project, but its detection heuristics (emulator fingerprinting alongside mock-location checks) are worth reading as an additional idea — e.g. also flagging capture happening on an emulator/rooted device, which is a related integrity signal beyond GPS alone. |

No dedicated, well-maintained "GPS spoofing detector for Android" library exists at the scale of, say, ZXing or Tink — this is a genuinely thin ecosystem, which is worth stating plainly in the report rather than overstating the maturity of what exists. The two `MockLocationDetector` repos above are small, low-star, but functionally real and directly portable.

## Final Pick List

One decisive recommendation per category — not a menu of maybes:

1. **Tamper-evident capture reference architecture → ProofMode** (`guardianproject/proofmode-android`, GPL-3.0). Study and adapt its capture→hash→sensor-snapshot→sign pipeline as the direct architectural model for Reality Lock's capture module.
2. **Content provenance standard → c2pa-android** (`contentauth/c2pa-android`). If the team has bandwidth, sign proof packages as real C2PA manifests via this Kotlin AAR rather than a bespoke JSON format — it's a Gradle dependency away and buys real interoperability.
3. **Crypto library → Google Tink** (`com.google.crypto.tink:tink-android`, Apache-2.0). Safer defaults than raw JCA, simpler API surface than Bouncy Castle, for a 2-student/1-semester team.
4. **Blockchain anchoring → OpenTimestamps** (`java-opentimestamps` from a Kotlin/Java backend, or `opentimestamps-client` from a Python backend), not web3j. Anchoring a hash to Bitcoin via free calendar servers requires no smart-contract deployment, no testnet faucet, no gas management — the realistic choice for the "future blockchain integration" goal on a semester timeline. Revisit web3j only if the project later specifically needs a custom on-chain registry contract.
5. **GPS integrity → MockLocationDetector** (`auag0/MockLocationDetector`, MIT). Port its four-check pattern (AppOpsManager scan, `Settings.Secure` flag, `Location` mock-flag inspection, basic hook detection) into a `LocationIntegrityChecker`, layered on top of `isFromMockProvider()`.
6. **QR / verification UI → ZXing via `zxing-android-embedded`** (Apache-2.0). One-dependency QR generation for the verification badge/certificate.
7. **PDF/certificate generation → Android's built-in `PdfDocument` API.** No new dependency, no licensing question, generates the certificate entirely on-device — the right amount of engineering for an MVP. Fall back to **OpenPDF** only if certificate generation is deliberately moved server-side later.
8. **Firebase alternative → Supabase** (Apache-2.0). Postgres gives real relational queries over proof packages (by location/time/user) that Firestore makes awkward, with a generous free tier and an actively maintained Kotlin SDK.
9. **Closest prior art to study before designing → Numbers Protocol's Capture Cam** (`numbersprotocol/capture-cam`) for the end-to-end product shape, alongside **ProofMode** for the capture-signing internals — the two together cover both halves of what Reality Lock is trying to build.
